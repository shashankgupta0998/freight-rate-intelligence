# Phase 6 — Deploy to Streamlit Cloud Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the FreightIQ stack publicly reachable at a Streamlit Cloud URL by un-gitignoring `requirements.txt`, regenerating it from `uv.lock`, configuring the Cloud app + secrets, smoke-testing the live URL, and updating CLAUDE.md with the deploy URL + redeploy workflow.

**Architecture:** No new code beyond a 1-line `.gitignore` change and a regenerated `requirements.txt`. Streamlit Cloud reads `requirements.txt` from the GitHub repo, installs deps, and runs `streamlit run app.py`. Secrets are configured in the Cloud dashboard (4 API keys + 3 flags) and Streamlit Cloud injects them to `os.environ` at app-process start. The existing `python-dotenv.load_dotenv()` calls become no-ops on Cloud (no `.env` file).

**Tech Stack:** `uv` (export), Streamlit Cloud (free tier), GitHub (already wired). No new Python packages.

**Source spec:** `docs/superpowers/specs/2026-04-24-phase6-deploy-design.md`

**Pre-flight:** Phases 1–5 shipped, all 96 tests passing, GitHub repo `shashankgupta0998/freight-rate-intelligence` is on `main`. `uv` installed locally.

## Task 3: Verify `requirements.txt` content meets spec acceptance

**Files:** none — verification only.

- [ ] **Step 1: Verify production deps are present**

```bash
echo "=== Required production deps (must all be present) ==="
for dep in streamlit langchain langchain-core langchain-litellm litellm pydantic requests python-dotenv beautifulsoup4 lxml; do
  if grep -iq "^${dep}==" requirements.txt; then
    echo "  ✓ ${dep}"
  else
    echo "  ✗ MISSING: ${dep}"
  fi
done
```
Expected: all 10 lines start with `✓`. If any line shows `✗ MISSING`, STOP and investigate the `uv export` output — likely a typo'd flag.

- [ ] **Step 2: Verify dev deps are EXCLUDED**

```bash
echo "=== Dev deps must NOT appear ==="
for dep in pytest pytest-cov coverage; do
  if grep -iq "^${dep}==" requirements.txt; then
    echo "  ✗ LEAKED: ${dep} (should be dev-only)"
  else
    echo "  ✓ ${dep} not present"
  fi
done
```
Expected: all 3 lines start with `✓ ... not present`. If any show `✗ LEAKED`, the `--no-emit-project` flag may not be filtering dev groups; switch to `--no-dev` if needed.

- [ ] **Step 3: Verify the project itself isn't pinned**

```bash
echo "=== Project self-reference must NOT appear ==="
if grep -iq "^freight-rate-intelligence==" requirements.txt; then
  echo "  ✗ LEAKED: freight-rate-intelligence==... (would break Streamlit Cloud)"
else
  echo "  ✓ project self-reference not present"
fi
```
Expected: `✓ project self-reference not present`. If it leaked, the `--no-emit-project` flag wasn't accepted by your `uv` version — fall back to `uv export --no-hashes | grep -v "^freight-rate-intelligence==" > requirements.txt`.

- [ ] **Step 4: Verify exact `==` pinning (no ranges)**

```bash
echo "=== Should be all == pins, no >= or ~= ==="
grep -E "^[a-z]" requirements.txt | grep -vE "^[a-z][^[:space:]]*==" | head -5
```
Expected: empty output (every dep line uses `==`). If any lines appear, they have non-`==` operators which is fine for pip but worth flagging — it means `uv.lock` didn't pin those.
